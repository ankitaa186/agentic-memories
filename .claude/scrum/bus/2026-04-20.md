## [00:42] PARMINDER -> TEAM: [DECISION] Epic 1 tech-spec posted
PARMINDER: Spec at `.claude/scrum/docs/tech-specs/epic-1-deep-profile-spec.md`. Top three design calls the team should know about:

1. **Confidence derives from `profile_sources`, not an incremented counter.** One source of truth, survives backfill, survives concurrent ingestions. New `ProfileConfidenceService` owns the formula (0.30 freq + 0.25 recency + 0.25 explicitness + 0.20 diversity, matches the column comment from migration 011). Called from `store_profile_extractions` after sources are inserted.

2. **Relationships is a new category on the existing `profile_fields` schema, NOT a new table.** `field_name = "{role}:{slug}"` (e.g. `daughter:ava`), `field_value` is `value_type='dict'` JSON. Reuses dedup, sources, confidence, and UI generic render path for free. Migration 026 just widens `chk_category_valid` (same pattern as 023).

3. **History is app-side, not a trigger.** `profile_fields_history` table written by a new `ProfileHistoryService` with change_type classification (`initial` | `refine` | `contradict` | `manual_override`). App-side because feature-flag rollout needs toggles and contradiction classification needs Python semantics. Backfill seeds one `initial` row per existing field, no fabricated history.

**Splits for Disha's epic:**
- "Wire confidence writes" should be TWO stories: service + call-site (forward-only writes) AND the backfill script. They deploy at different times.
- "Relationships" should be THREE stories: (a) migration + category + VALID_CATEGORIES, (b) prompt + extraction path, (c) UI render path (icon + dedicated relationship card). Different reviewers, different risk profiles.
- "Temporal decay/freshness" and "Contradiction detection" should stay separate stories — they share the history table but serve different flows.
- "Prompt additions for new flat fields" should name every new field in its acceptance criteria so Murat can write per-field extraction regression tests.

**Flag matrix (six independent env vars):** `PROFILE_WRITE_CONFIDENCE`, `PROFILE_LOG_HISTORY`, `PROFILE_ENABLE_RELATIONSHIPS`, `PROFILE_NEW_FIELDS_COUNT`, `PROFILE_CONTRADICTION_DETECTION`, `PROFILE_TEMPORAL_DECAY`. Each story lands dark, each flag flips live when ready. The NEW_FIELDS_COUNT flag is load-bearing — flipping new fields into `EXPECTED_PROFILE_FIELDS` without gating tanks every existing user's completeness percentage.

**Calls I deferred (need user/Disha input):**
- Manual PUT semantics — I left `confidence=100` as-is for Epic 1. Changing it mid-epic risks surprising existing callers. We expose `is_manual_override` in the read path instead. Revisit in Epic 2?
- Retention on `profile_fields_history` — "keep forever" for now.
- `recent_wins` decay — ship uniform decay in R7, tune with data rather than adding a `decay_profile` column now.

**Ordered rollout (R1 → R8):** migrations + backfill scripts → confidence service → history service → new flat fields → relationships → contradiction surfacing → temporal decay → UI. R2/R3/R4/R5 parallelize after R1.

No blockers identified. All changes additive or flag-gated. Existing `/v1/profile` response shape byte-identical by default; new data flows through opt-in query flags.
