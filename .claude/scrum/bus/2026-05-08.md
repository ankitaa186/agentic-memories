Surprises / notable decisions:
- The Pydantic sentinel for `ttl_seconds` ended up needing `Any`-typed field (not `Optional[Union[int, _Unset]]`) because Pydantic v2 validates field types and would reject the sentinel-class instance. `Any` + `default=UNSET` lets the field pass through `null` as `None` and absent-key as the default singleton. Confirmed survives JSON round-trip via `model_validate_json('{}')` → UNSET vs `model_validate_json('{"ttl_seconds": null}')` → None (test added in `test_memories_patch.py`).
- AC12 reads "importance (→ importance_score)" but ONLY `episodic_memories` has an `importance_score` column. `emotional_memories` and `procedural_memories` do not. I made the `_update_emotional_row` / `_update_procedural_row` signatures NOT accept `importance` — when the caller patches importance on a record that fans out to those tables, the value still lands in Chroma metadata (which is the source of truth) but is silently ignored on the typed-table side. This avoids generating invalid SQL while still respecting the patch.
- `update_chroma_record` takes both `metadata` (caller-supplied, gets stripped) and `internal_metadata` (server-computed, NOT stripped) so the router can write `ttl_epoch`, `content_hash`, `layer`, `timestamp` etc. without bypassing the AC20 protection for caller-domain keys. Internal metadata wins on key collisions.
