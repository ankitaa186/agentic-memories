# Disha — PM Memory: agentic-memories

## Project in one paragraph
Agentic memory system for Ankit (Telegram user 1169491032). Captures memories across typed stores (episodic, emotional, portfolio) with vector + keyword hybrid retrieval. On top of raw memories sits a **profile layer** — a structured self-model extracted from memories via LLM. FastAPI + React/TS UI; Docker-composed stack. Primary user value: an AI that knows you well enough to be genuinely useful, not just retrieve.

## Who the user is (from a PM lens)
Single primary user, target Ankit. This is a personal intelligence system, not a multi-tenant SaaS — which means we optimize for depth and quality of a single profile rather than scale. The user is also the builder.

## The product frame I'm working from
**"Resume vs self-model."** Today the profile stores name, age, occupation, hobbies — the facts you'd put on a business card. What's missing: the present-tense (where am I in life right now), the relational (who matters to me and how), the epistemic (how sure are we about any of this), and the temporal (is this current or stale). Epic 1 closes those four gaps.

## Current epic: Epic 1 — Deep Profile (v2)
Written to `.claude/scrum/docs/epic-1-deep-profile.md`. 19 stories across P0/P1/P2. Addresses the 7 audit priorities plus health/semantics cleanup plus category re-assignment.

## Big shift from v1 to v2 — 2026-04-20 [01:32]

User relaxed the constraint. v1 was written against "don't break anything." v2 is written against **API backward-compat only** — URL paths, HTTP methods, status codes, response top-level keys, query params, PUT/DELETE body shapes are the fence. Stored data, `EXPECTED_PROFILE_FIELDS`, `completeness_pct` math, UI, category assignments are all free to change.

### What that let me do
- **Retire `depth_completeness_pct` entirely.** In v1 I was protecting the 27-field baseline from drift by tracking new fields in a parallel signal. v2: one number, one signal, recomputes over the new expected set. Stories 1.7, 1.10, 1.11 all got simpler.
- **Collapse the flag matrix from 7 → 3.** Kept `PROFILE_WRITE_CONFIDENCE` (shadow-verify the scoring formula), `PROFILE_CONTRADICTION_DETECTION` (observe before resolve), `PROFILE_INFERRED_EXTRACTION` (LLM prompt change worth dark-launching). Retired `PROFILE_LOG_HISTORY`, `PROFILE_ENABLE_RELATIONSHIPS`, `PROFILE_TEMPORAL_DECAY`, `PROFILE_NEW_FIELDS_COUNT`, `PROFILE_DEPTH_FIELDS_ENABLED` — just ship them.
- **Turn the semantic cleanup stories into destructive migrations.** 1.13 (dietary), 1.14 (allergies null-vs-empty), 1.15 (health_conditions string→object) no longer hide behind shadow fields; they fix data in place with `profile_fields_history` as the audit trail.
- **Add Story 1.19 — category re-assignment.** Previously outside scope because it would have required "don't change what category a field lives in." v2 lets me move fitness content from `interests.hobbies` → `health.exercise`. Concrete vocabulary list documented in the story.
- **Tighten every story's ACs with "API contract preserved" explicitly naming response keys.** This was implicit in v1 ("non-breaking notes"); v2 makes it first-class because the contract IS the only remaining fence.

## What I learned from the codebase (still valid)
- `profile_fields` is schemaless per-field with JSONB values. New categories and fields add without schema migration as long as `VALID_CATEGORIES` accepts them.
- `profile_confidence_scores` table has the 4-component schema already; wiring is a service-level change.
- `profile_sources` already populated; confidence derives from it directly.
- The `TOTAL_EXPECTED_FIELDS = 27` comment lies — it's actually 27 today; in v2 it grows. Story 1.1 + regression test guards against drift.
- Profile.tsx iterates whatever the API returns; in v2 it gets redesigned (Story 1.18) because UI is no longer frozen.

## Key product decisions carried into v2
1. **Contradiction detection before resolution.** Story 1.4 observes; 1.5 resolves. Still worth splitting — ship observation, learn from real oscillation data, then commit to a resolution algorithm.
2. **Inferred extraction is flag-gated.** Genuine LLM behavior change; worth a dark-launch even when the rest shed their gates.
3. **Every destructive migration produces a history row.** `change_type='migration'` becomes the audit trail for the four destructive stories. This is also why 1.6 (history table) is a hard dependency for 1.13/1.14/1.15/1.19.
4. **Success metrics reframed around user-observable outcomes.** Not "flag is on" but "conflicts surfaced", "freshness visible", "depth captured". Forces me to think about whether the feature is actually delivering, not whether the plumbing exists.
