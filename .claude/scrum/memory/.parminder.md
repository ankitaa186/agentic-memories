# Parminder — Project Map (agentic-memories)

_Last updated: 2026-04-20 (v2 tech spec — constraint relaxed to API-backward-compat only)_

## Tech stack (architecture lens)

- **Backend**: FastAPI, Python. Entrypoint `src/app.py`. Routers in `src/routers/`, services in `src/services/`, dependencies in `src/dependencies/`.
- **Data stores**:
  - ChromaDB (tenant=`agentic-memories`, db=`memories`, collection=`memories_3072`) — vector memory.
  - TimescaleDB — structured tables (20 tables, 3 hypertables: `episodic_memories`, `emotional_memories`, `portfolio_snapshots`). Port 5432 (old setup used 5433).
  - Redis — caches (completeness, activity tracking).
- **Frontend**: React + TypeScript + Vite + TanStack Query, served via nginx on 3003. API on 8080.
- **Ingestion orchestration**: `src/services/unified_ingestion_graph.py` — LangGraph DAG with nodes for dedup, extract_profile, store_profile, store_chromadb, etc.
- **Migrations**: `migrations/postgres/NNN_description.{up,down}.sql`, applied by `migrations/migrate.sh`. Currently at 024.

## Key modules (profile domain)

- `src/services/profile_extraction.py` — LLM-driven extraction. Single `PROFILE_EXTRACTION_PROMPT` defines canonical field names. Has `_deduplicate_extractions` and `_validate_extractions`. Knows `FIELD_NAME_ALIASES`.
- `src/services/profile_storage.py` — Writes to `profile_fields`, `profile_sources`, `user_profiles`. Owns `EXPECTED_PROFILE_FIELDS` (source of truth for completeness; currently 27 fields across 8 categories). **Does NOT currently write to `profile_confidence_scores`** — that's Epic 1's work.
- `src/routers/profile.py` — REST at `/v1/profile`. Manual PUT at lines 305-452 does write `profile_confidence_scores` with all-100 values.
- `ui/src/pages/Profile.tsx` — Renders profile generically. `categoryIcons` + `categoryColors` maps at lines 29-49 must be extended for new categories.

## Tables touched by profile

- `user_profiles` (migration 009) — 1-row-per-user metadata; completeness_pct, field counts.
- `profile_fields` (010) — `(user_id, category, field_name)` PK. TEXT field_value + value_type discriminator. `chk_category_valid` constraint (most recently expanded in migration 023).
- `profile_confidence_scores` (011) — **already has the full column set**: overall + 4 components + mention_count + last_mentioned. Only written from manual PUT today.
- `profile_sources` (012) — append-only audit of every extraction. `source_type IN (explicit, implicit, inferred)`.

## Patterns I'm seeing

1. **Additive migrations** — 023's constraint-widening is the template to follow when extending `chk_category_valid`. Drop + re-add with superset.
2. **Feature flags = env booleans** via `config.py`. No runtime DB-config system yet. Keep that simple.
3. **Dedup inside a single ingestion batch** happens at the service layer (`_deduplicate_extractions`). Cross-batch aggregation is via row-level primary keys.
4. **Field `value_type='dict'`/`list`** is the path for structured values; JSON serialized into a TEXT column. Works fine today — I used this for the relationships category in the Epic 1 spec.
5. **Completeness caching** in Redis with invalidation on any profile write. Cache key: `profile_completeness:{user_id}`, TTL 1h.

## Known tech debt / smells

- `# 25` comment at `profile_storage.py:39` is stale — actual `TOTAL_EXPECTED_FIELDS` is 27.
- `_identify_high_value_gaps` at `profile_storage.py:457` has a weird early-return structure and a redundant conditional (`category == "basics"` branch that does nothing).
- `profile_confidence_scores` being dead code in the auto path is the biggest smell — schema existed from migration 011, never populated.
- Manual PUT forces confidence to 100 and increments `mention_count`. Arguably that's incorrect semantics (manual is not 5 mentions worth of evidence), but it's the contract today.
- Migration 024 cleaned up double-encoded JSON string values — this happened because LLMs occasionally returned JSON-encoded strings. The `_validate_extractions` now defends against it, but we should assume occasional historical data weirdness persists.

## Risk map for Epic 1 (v2)

- **Biggest risk now: destructive data migrations on production data.** Four migrations rewrite existing rows (dietary de-conflation, health_conditions string→object, allergies null-vs-empty, category reassignment). Mitigated by idempotency contract, dry-run mode, per-user transactional wrap, and a recommended point-in-time backup before first prod run.
- **Second biggest: prompt regression from adding too many fields at once** (relationships + 7 flat fields + inferred guidance). Test coverage must grow with the prompt. `PROFILE_INFERRED_EXTRACTION` is the only flag that dark-launches this for comparison. Murat owes a per-field extraction regression suite before R2 flips.
- **Third: completeness score visibly drops the day R1 ships** because denominator grows from 27 → 35. User-accepted; release-note material.
- **Concurrency (unchanged):** two ingestions racing on confidence recomputation — tolerable because formula is deterministic from `profile_sources`.
- **Gotcha:** `migrations/migrate.sh` defaults `PG_PORT=5433` but current setup uses 5432. Pre-flight check before running destructive migrations 027–030.
