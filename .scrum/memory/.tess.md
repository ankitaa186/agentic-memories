# Murat's Project Understanding (Tester)
Last Updated: 2026-05-08 00:35

## Test Infrastructure
- Framework: **pytest 8.3.2**
- Config: `pytest.ini` at repo root
- Discovery: `testpaths = tests`, `test_*.py`, `Test*` classes, `test_*` functions
- Markers: `unit`, `integration`, `e2e`, `slow`
- Default opts: `-v --tb=short -ra`
- Top-level shared `conftest.py`: `tests/conftest.py`
- Per-suite conftests in `tests/integration/`, `tests/e2e/`, `tests/memory_orchestrator/`

## Test Suites at a Glance
- `tests/unit/` — ~17 modules: app health/helpers/retrieve/store, condition expression validation, cooldown logic, fire mode, intent validation/schemas, memories router, next-check calculation, portfolio API/trigger, profile API/completeness, timezone validation
- `tests/integration/` — `test_direct_memory_api.py`, `test_intents_api.py` (mocked deps)
- `tests/e2e/` — full E2E against running stack (`test_e2e_api.py`, `test_e2e_comprehensive.py`, `test_e2e_evaluation.py`, `test_intents_e2e.py`); has its own `run_e2e_tests.sh` and `README.md`
- `tests/memory_orchestrator/` — orchestrator-specific suite
- `tests/evals/` — eval harness (likely Langfuse-backed); excluded by `norecursedirs`
- `tests/test_orchestrator_api.py` — top-level orchestrator API test

## Run Commands
- `make test-unit` — fastest, no deps
- `make test-integration` — mocked
- `make test-e2e` — requires running Docker stack (`make start` first)
- `make test-fast` — unit + integration
- `make test` — everything
- `make test-coverage` — coverage report
- `make test-intents`, `test-intents-e2e`, `test-memory`, `test-profile` — domain-scoped
- `tests/e2e/run_e2e_tests.sh` — for ad-hoc e2e

## Coverage Gaps / Red Flags
- **Three prompt modules** (`prompts.py`, `prompts_v2.py`, `prompts_v3.py`) — unclear which is exercised. Likely a coverage hole worth investigating.
- **Eval suite is excluded from default discovery** (`norecursedirs`). Make sure new extraction work updates evals, not just unit tests.
- **Repo-root `test_profile_extraction.py`** — outside `tests/`, may not be picked up by pytest discovery. Confirm before relying on it.
- **`tests/e2e/CLEANUP_SUMMARY.md`, `IMPLEMENTATION_SUMMARY.md`, `TEST_ANALYSIS.md`** — historical analysis docs; useful context before writing new e2e tests but not authoritative current state.
- **No mention of property-based tests / Hypothesis** in deps — if extraction shape robustness becomes a theme, propose Hypothesis.
- **Pydantic silent-drop** is a cross-cutting test category. Murat's job: every new schema field gets a "produces this exact shape" producer test.

## Testing Philosophy for This Project
- **Shift-left**: review acceptance criteria during the `drafted` -> `ready` transition; flag missing test cases to Disha/Parminder before David starts coding.
- **Producer + consumer tests**: for any fail-closed validation, write tests on both sides.
- **Don't mock the database in integration tests** unless absolutely needed — the user has been burned by mock/prod divergence (cross-project lesson).
- **Coverage isn't the goal, scenario coverage is**: ACs map to test cases, edge cases get explicit tests.
