## Patterns I'm seeing

1. **Additive migrations** ŌĆö 023's constraint-widening is the template to follow when extending `chk_category_valid`. Drop + re-add with superset.
2. **Feature flags = env booleans** via `config.py`. No runtime DB-config system yet. Keep that simple.
3. **Dedup inside a single ingestion batch** happens at the service layer (`_deduplicate_extractions`). Cross-batch aggregation is via row-level primary keys.
4. **Field `value_type='dict'`/`list`** is the path for structured values; JSON serialized into a TEXT column. Works fine today ŌĆö I used this for the relationships category in the Epic 1 spec.
5. **Completeness caching** in Redis with invalidation on any profile write. Cache key: `profile_completeness:{user_id}`, TTL 1h.

### Embedding cost = 0 ŌĆö actual root cause (verified)
**Langfuse `_parse_usage` for embeddings emits `{"input": prompt_tokens}`** at `langfuse/openai.py:564-570`. The seeded `text-embedding-3-large` model definition in `worker/src/constants/default-model-prices.json` (Langfuse repo, line 798-817) has price keys keyed on `"total"` ONLY:
```json
"prices": { "total": 1.3e-7 }
```
Langfuse Cloud's cost-calc requires usage_type ŌåÆ price_key exact match. `input` ŌēĀ `total`, so multiplication is skipped ŌåÆ cost=0. This isn't our bug ŌĆö it's a wrapper/seed data mismatch. Same applies to ALL embedding models in their seed file (3-small, ada-002 use `total` too).

**Fix paths (in order of preference):**
1. Override `text-embedding-3-large` in our Langfuse project with a custom model definition that has `prices: { input: 1.3e-7 }` (or add `total: 1.3e-7` to mirror). Zero code change. ~5 min in Langfuse UI.
2. Compute embedding cost ourselves and `update(usage_details={"input": N, "output": 0, "total": N})` after the call.
3. Upstream PR to either langfuse-python (emit `total` for embeddings) or langfuse seed file (add `input` key). Slow.

### Chat cost ŌĆö `gpt-5.2` model name (verified)
**Wrong before:** I claimed `gpt-5.2` was not a real OpenAI model. It IS real (released 2025-12-11; OpenAI docs at `developers.openai.com/api/docs/models/gpt-5.2`). `.env` value is correct.

Pre-condition for chat: `_parse_usage` returns `prompt_tokens`/`completion_tokens` and Langfuse remaps to `input`/`output` (per their docs).

## Worthiness duplication (added 2026-04-25, post-22:42)

### Symptom
User's Langfuse trace export from a non-worthy `/v1/store` showed TWO `OpenAI-generation` observations inside the `worthiness` chain, back-to-back, ~3s each, identical prompt/input. Same parent, same depth, auto-generated `OpenAI-generation` name on both ŌåÆ not a deliberate dual-model voting setup, both came through the same SDK code path.

### Root cause (high confidence)
`src/services/extract_utils.py:117-146` ŌĆö `_call_llm_json` wraps the `client.chat.completions.create(...)` call in `for _ in range(retries + 1):` with `except Exception: last_exc = exc; continue`. `get_extraction_retries()` (config.py:191-196) defaults to `1` ŌåÆ loop runs up to **2 attempts**. Any exception out of the first attempt ŌĆö transient network error, `IndexError` on `resp.choices[0]`, decode error in the SDK ŌĆö triggers a brand-new `client.chat.completions.create(...)`. The `langfuse.openai._wrap` (`langfuse/openai.py:814-881`) creates exactly one observation per `wrapped()` invocation, so two SDK calls ŌåÆ two `OpenAI-generation` observations in the trace. The first one would be marked `level=ERROR` with `status_message` (line 874-879) ŌĆö the export snippet didn't include those fields.

### What this is NOT
- Not a langgraph node binding bug (verified `unified_ingestion_graph.py:1317` adds `worthiness` once).
- Not a `should_continue` / tools_condition loop (no such edge ŌĆö worthiness goes straight to `decide_after_worthiness` conditional).
- Not the langfuse SDK double-emitting (`_wrap` is a single observation per call ŌĆö confirmed by reading `_wrap` body).
- Not OpenAI SDK internal retries (`DEFAULT_MAX_RETRIES=2`) ŌĆö those happen inside one `create()` call and roll up to one observation.
- Not a LangChain `with_retry` / `with_fallbacks` wrapper ŌĆö `grep` shows no such usage in `src/`.
- Not structured-output reparse ŌĆö `response_format={"type": "json_object"}` is a single-call hint, no SDK-level reparse loop in 1.40.0.

### Blast radius
`_call_llm_json` is called by 8+ sites: worthiness, extraction, sentiment, profile_extraction, graph_extraction, compaction, reconstruction, app.py. So whenever `EXTRACTION_RETRIES > 0` AND the first call raises, EVERY downstream consumer pays double latency + double cost (once cost works). The trace just happened to surface it on worthiness because that's the cheapest call and the duplication is most visually obvious there.

### Smallest fix (proposed; not yet a story)
- `extract_utils.py:144` (and again in xai branch at line 191): replace `except Exception` with a narrower set: `httpx.TimeoutException`, `httpx.ConnectError`, `openai.APIConnectionError`, `openai.RateLimitError`, `openai.APITimeoutError`, `openai.InternalServerError` (5xx). Anything else (IndexError, KeyError, JSON decode, prompt validation) should bubble up ŌĆö those are deterministic and a retry just doubles cost.
- Add a `logger.warning("LLM call attempt N/M failed: %s ŌĆö retrying", attempt, exc)` inside the except block so future duplicate-call incidents are debuggable from our own logs without a Langfuse trip.
- ~15-20 LOC, single file, no API surface change.

### Verification path before shipping fix
Look at the user's actual trace in Langfuse UI: confirm the FIRST `OpenAI-generation` has `level=ERROR` and a `status_message`. If yes ŌåÆ diagnosis confirmed. If no (both `level=DEFAULT`) ŌåÆ the retry isn't from our `except` and we need to revisit (possible langfuse SDK regression we missed, or an httpx-level retry).

### Story shape
Single P1 story (one helper, one bug, one localized fix). Test plan for Murat: unit test that monkeypatches `client.chat.completions.create` to raise `IndexError` (non-retryable) and asserts the helper raises/returns None WITHOUT a second call; counterpart test that raises `openai.APITimeoutError` and asserts a second call IS made. Out of Epic 1 (still on hold). Companion to Story 2.1 (embedding entry-point wrapping); both belong in Epic 2 (Observability) if Disha opens one.

### Verified `openai==1.40.0` exception classes (uv.lock + `.venv/.../openai/_exceptions.py`)
- All six classes Disha named exist in `openai` namespace.
- `APITimeoutError` is a subclass of `APIConnectionError`. Both listed in narrow tuple for intent/readability.
- `APIError` is base class for all openai errors INCLUDING 4xx client errors ŌĆö DO NOT add to tuple.
- `UnprocessableEntityError` (422) is non-retryable client error ŌĆö DO NOT add.
- xAI uses the same `OpenAI(api_key=..., base_url=...)` client ŌåÆ same exception classes apply.
- `httpx` errors normally get wrapped by the SDK before exit, so `httpx.TimeoutException` / `httpx.ConnectError` entries are belt-and-suspenders ŌĆö keep them anyway (cheap defense).

### Smell flagged for the future
The outer `except Exception ŌåÆ return None` at `extract_utils.py:200-220` is the second silent-failure layer. Today it's net-positive (traceback logging + Langfuse `trace_error` breadcrumb). But "return None on any unexpected failure" makes pipeline failures invisible to callers (they see "no extractions today" instead of "the LLM is broken"). Worth revisiting AFTER Story 2.1 lands ŌĆö once we can see error spans natively in Langfuse, the outer except's value drops and we may want exceptions to propagate. Track as a future Epic 2 candidate, NOT a 2.2 blocker.
